## 一休的核心原則（寫死，挑戰時以此為本）

1. **意願判定二分**（continue / decline，沒 Medium）+ **按鈕 = 明確意願**（點了才進 /apply，沒點不推銷）
2. **Q5 訊息不講價格**：方案細節留給 /apply 頁（避免跟 Q4 敘事重複 / 避免推銷感）
3. **追蹤「點了 /apply 卻沒報名」是核心 KPI**
4. **真人保留處理猶豫的 20%**（handoff 訊息要承諾具體時間 + 表達「不急推課」）

**影響範圍**：既有 admin 後台 / drip / push 都傳 `{label, url}` 格式 → 加 `type: 'uri'` default 保持向下相容。

## 3. AI 輸出契約

### 3.1 Q5_intent_classifier（唯一 AI，判 continue/decline）

**觸發**：stage=4 用戶收到文字訊息，且不命中 polite end / handoff 關鍵字，長度 ≥ 2 字。

**輸入**：
- `{user_text}`：用戶回覆

**Prompt**（英文 key）：
```
You are analyzing a user's response after receiving personalized weight-management feedback from Yixiu (a Taiwanese weight coach).

User's reply: "{user_text}"

Classify the user's intent:
- "continue" = User shows interest, wants to know more, or responds positively (including short ack like "好", "嗯", "了解", "真的嗎", "有道理", "不錯")
- "decline" = User explicitly rejects, not interested, wants to think later, or expresses negative feeling toward the feedback

Return JSON only:
{"intent": "continue" | "decline", "confidence": "high" | "medium" | "low", "reason": "<one sentence why>"}

Examples:
- "好 我想知道更多" → {"intent":"continue","confidence":"high","reason":"明確表達想了解"}
- "嗯嗯" → {"intent":"continue","confidence":"medium","reason":"有回應無否定"}
- "這些我都知道了" → {"intent":"decline","confidence":"medium","reason":"表達不需要更多"}
- "讓我想一下" → {"intent":"continue","confidence":"low","reason":"猶豫但未拒絕，推軟邀請不會騷擾"}
- "其實我還沒準備好" → {"intent":"decline","confidence":"high","reason":"明確表達尚未準備"}
```

**Fallback（v2 嚴進，洞 3 修）**：
- AI timeout / JSON 無效 → **不推 Q5，保持 stage=4 靜默**（v1 是寬進，v2 改嚴進避免誤推）
- 但 confidence='low' 的 continue **照推**（因為 Q5 訊息本身是軟邀請，按鈕才是意願 filter）
- 完全沒回應（7 天）→ 視為 decline（cron 不推）

**Validator（Phase 4.2 必寫）**：
```js
function validateQ5Intent(output) {
  if (!output || typeof output !== 'object') return { valid: false };
  if (!['continue', 'decline'].includes(output.intent)) return { valid: false };
  if (!['high', 'medium', 'low'].includes(output.confidence)) return { valid: false };
  if (typeof output.reason !== 'string') return { valid: false };
  return { valid: true, data: output };
}
```

**寫入**（Ch.0.3 updatePathStage 擴充後）：
```js
await updatePathStage(userId, 6, {
  q5_sent_at: now,
  q5_intent: 'continue',
  q5_classified_at: now,
});
```

### 3.2 Q5 軟邀請訊息（靜態模板，v2 砍 AI）

**決策 A 改動**：Q5 訊息不再由 AI 產，用靜態模板填變數。理由：
1. Q4 AI 已給個人化敘事，Q5 再 AI 會重複痛點（Pattern 1/3）
2. 省 Gemini call
3. 方案細節放 /apply 頁，LINE 訊息不推銷

**Flex Message 結構**（靜態，pushFlexMessage 擴 postback 後可直接組）：

```json
{
  "type": "bubble",
  "body": {
    "type": "box",
    "layout": "vertical",
    "contents": [
      {"type": "text", "text": "想瘦 {diff} 公斤這個目標我們有完整課程", "weight": "bold", "size": "md", "wrap": true},
      {"type": "text", "text": "4 週體驗、12 週完整兩種方案都有。要不要看一下適合你的？", "wrap": true, "margin": "md", "size": "sm"}
    ]
  },
  "footer": {
    "type": "box",
    "layout": "vertical",
    "spacing": "sm",
    "contents": [
      {
        "type": "button",
        "action": {"type": "uri", "label": "看看方案", "uri": "https://official-yihugh-line-bot.vercel.app/apply?userid={userId}&source=bot_q5"},
        "style": "primary",
        "color": "#5B8DEF"
      },
      {
        "type": "button",
        "action": {"type": "postback", "label": "有問題想問", "data": "action=handoff_from_q5"},
        "style": "secondary"
      }
    ]
  }
}
```

**變數**：`{diff}` / `{userId}` — 只有兩個，極少 NULL 風險。

**NULL fallback**：
- `{diff} < 1` 或 NULL → 用「你瘦身目標」取代（保底模板）
- `{userId}` 一定有（webhook event.source.userId 強制存在）

**禁止**：
- 訊息內**不提價格**（省 Q4 重複 + 讓 /apply 頁的方案介紹有資訊價值）
- 訊息內**不重述痛點**（Q4 已做）
- **不列點、不長篇大論**（短邀請 < 60 字）

## 5. 觸發規則表

### 5.1 Stage=4 → Stage=6（推 Q5 軟邀請）

```
觸發：stage=4 用戶傳文字訊息（message.type === 'text'）
條件：
  1. 不命中 polite end 關鍵字（handoff.js:matchPoliteEnd）
  2. 不命中 handoff 關鍵字（handoff.js:matchGlobalHandoff）
  3. 文字長度 ≥ 2 字
動作：
  1. 呼叫 Q5_intent_classifier → intent
  2. 若 intent='decline':
     → 寫 q5_intent='decline', q5_classified_at=now（不動 stage）
     → 靜默（或視情況回 polite end 既有話術）
  3. 若 intent='continue'（含 confidence=low）:
     → updatePathStage(userId, 6, { q5_sent_at, q5_intent:'continue', q5_classified_at })
     → pushFlexMessage(userId, 軟邀請 Flex)（靜態模板 + {diff} + {userId}）
  4. 若 AI timeout / JSON invalid:
     → 保持 stage=4 靜默（v2 嚴進 fallback）
下一狀態：stage=6 / stage=4 維持
```

### 5.2 Stage=6 → Stage=7（點 /apply 按鈕）

```
觸發：用戶在 LINE 點 Flex「看看方案」→ 瀏覽器開 /apply?userid=X&source=bot_q5
實作：
  - /apply 是 Next.js server component（抓 userId query，pre-fill 表單）
  - page render 時，client 組件用 useEffect 發 POST /api/apply/visit {userId}
  - 用 sessionStorage key 'q5_visit_recorded_{userId}' 擋重複寫（同頁 refresh 不加 count）
條件（API 後端）：
  1. userId 存在於 official_line_users
  2. stage >= 6（防跳階）
動作：
  1. UPDATE stage=7（若當前 stage=6）, q5_clicked_at=now（first-write, COALESCE 保留原值），q5_click_count += 1
  2. 回 OK
下一狀態：stage=7（若原本 stage=6，stage=7+ 則保持不 regress）
```

### 5.3 Stage=7 → Stage=8（submit）

```
觸發：/apply 送出
API：POST /api/apply/submit {formData + userId + source}
條件（後端驗證）：
  1. 必填齊全（real_name / phone / email / address / gender / age / program_choice）
  2. agreed_refund_policy === true
  3. age ∈ [18, 99]
  4. phone 格式（09 開頭 10 碼）
  5. email 基本格式
動作（原子性改進，洞 11 修）：
  1. Supabase RPC 'submit_application' 或 TypeScript 單次呼叫含：
     a. INSERT applications (status='pending', source='bot_q5', notify_status='pending')
     b. UPDATE users (stage=8, enrolled_at=now if null, enrolled_from_path=path if null)
     → 兩者同 transaction
  2. Commit 後：await pushMessage 給婉馨 + 一休（notify_enrolled 模板）
  3. 若 notify 成功 → UPDATE apps.notify_sent_at=now, notify_status='sent'
  4. 若 notify 失敗 → UPDATE apps.notify_status='failed'（cron retry）
  5. 回前端 success 頁：「婉馨老師會在 24 小時內聯繫你確認匯款」
下一狀態：stage=8（終態）
```

### 5.4 Cron：Q5 跟進「點了沒送出」

```
觸發：/api/cron/q5-followup（合併 cron 的情境在 Ch.0.4 決定）
掃描：
  stage=7
  AND q5_clicked_at < now - q5_followup_hours (24h)
  AND enrolled_at IS NULL
  AND q5_followup_sent_at IS NULL
動作（每用戶）：
  1. pushFlexMessage（含按鈕 postback handoff_from_q5）
     訊息：「剛剛看了報名頁是有什麼想問的嗎？我請 fifi 直接跟你聊」
  2. UPDATE q5_followup_sent_at=now
  3. 不推進 stage（保持 7）
  4. 不重試（推一次夠）
```

### 5.5 Cron：reset 卡死（洞 4 + 洞 12 修）

```
觸發：/api/cron/stage-timeout（每日 03:00 UTC）
掃描（v2 基準改 q5_sent_at）：

  AND q5_sent_at < now - q5_timeout_reset_hours (48h)
  AND enrolled_at IS NULL
  AND q5_clicked_at IS NULL  -- 沒點過（保留 stage=7 用戶）

  1. UPDATE stage=4, q5_sent_at=NULL
  2. 保留 q5_intent / q5_classified_at（給統計）

  - 已訪問 = 用戶有意願行動，給他時間（或等 Ch.5.4 cron 跟進後若仍無動作由婉馨人工接手）
```

## 6. Handoff / Fallback 規則

### 6.1 Handoff 與 Q5 互斥

- 用戶發 **handoff 關鍵字** → stage=5（覆蓋 6/7）
- Q5 按鈕 2「有問題想問」postback → triggerHandoff(reason='q5_followup')
- Ch.5.4 cron 跟進按鈕「跟 fifi 聊」→ 同上

### 6.1.1 Stage=6/7 專屬 handoff 訊息（洞 7 修）

**現有 handoff 訊息**：「你這個問題值得好好聊，我請 fifi 助教私訊你，她會主動找你。」

**Stage=6/7 觸發 handoff 的專屬版本**：
```
我有看到你的問題。這個我請 fifi 直接跟你聊，她看過你剛剛跟我聊的內容，會知道你在哪個階段，等等會主動找你。

先不急著決定要不要進課程，把問題問清楚再說。
```

實作：`handoff.js:triggerHandoff` 根據 `reason` 或 `prev_stage` 選文案：
- `reason='q5_followup'` or `prev_stage ∈ {6,7}` → 專屬版本
- 其他 → 既有文案

### 6.2 Polite end 與 Q5 互動

- Q4 後用戶傳 polite end 關鍵字（太貴/沒預算/沒錢/先不用）→ 既有 handlePoliteEnd → Q5_intent 跳過
- Stage=6 用戶傳 polite end 關鍵字 → 同上，stage 保持 6（等 cron reset）

### 6.3 AI Fallback 明細（v2 嚴進）

| 情境 | Fallback 行為 |
|------|--------------|
| Q5_intent_classifier timeout / JSON 無效 | **保持 stage=4 靜默**（不推 Q5，v2 改嚴進）|
| pushFlexMessage 失敗 | `q5_sent_at=NULL`, stage 保持 4（下次觸發重試）|
| /apply/visit API 失敗（DB error） | 前端重試 3 次 + 寫 console.error，不擋用戶看頁面 |
| /apply/submit transaction 失敗 | 前端顯示「系統忙碌請稍候再試」，不 INSERT 不 UPDATE |

## 7. 多選 / 歧義處理 + 非文字訊息矩陣

### 7.1 文字訊息場景

| 場景 | 處理 |
|------|------|
| stage=6 用戶傳文字（非 CTA）| webhook dispatch：handoff 關鍵字 → stage=5；polite end → polite end；其他 → 靜默（避免插嘴）|
| 重複點 CTA | q5_click_count +=1，stage 保持（不 regress）|
| submit 兩次 race | 後端用 sessionStorage + DB 檢查 `last_submitted < 5s` 擋快速重送；兩筆都通過則 notify 提醒婉馨 |
| 填錯欄位被退 | 前端顯示錯誤，stage 保持 7 |
| stage=8 再傳訊息 | 靜默（未來 Phase 5 加「已報名」話術）|

### 7.2 非文字訊息矩陣（洞 6 修）

| stage | 訊息類型 | 處理 |
|-------|---------|------|
| 任何 | `image` | 既有 `handleImageMessage`（stage=3 引導回數字；其他 stage 靜默）|
| **6/7** | `image` / `sticker` / `video` / `audio` / `file` / `location` | **軟 handoff**：回「你的訊息我收到了，我請 fifi 直接跟你聊看看問題」+ triggerHandoff(reason='q5_non_text_query') |
| 其他 stage | 非文字訊息 | 維持現有行為（多數靜默）|

**實作**：webhook `handleEvent` 最前層加 stage 判斷，進非文字 handler 前先問「是不是 stage=6/7」。

### 7.3 Q5 軟邀請訊息本身被 LINE 表情回應（洞 6 衍生）

- LINE reaction（讚、愛心等）**不產生 webhook event**（LINE API 限制）
- 所以不需特別處理，繼續維持 stage=6

| 指標 | 分子 | 分母 | 排除 |

| q5_click_rate | DISTINCT line_user_id WHERE q5_clicked_at IS NOT NULL | q5_sent_count | 同上 |

- `q5_enrollment_rate ≥ 10%` 才算 Q5 有效
- `q5_block_rate ≤ 2%` 才算不擾民
