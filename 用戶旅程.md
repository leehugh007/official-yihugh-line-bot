## A 軌 → B 軌銜接鉤子（容易忘，寫死）

**關鍵事實**：A 軌的「菜單」「地雷」回覆 template **末尾會問**「告訴我你目前幾公斤、想瘦到幾公斤」。

這是 A 軌把用戶**主動推**到 B 軌 Q1-Q2 的設計 — **不是婉馨人工做的**，是 Bot 自動化的。

設計意圖：
- 用戶對「地雷 / 菜單」有興趣 = 有瘦身意願
- 趁熱問體重 → 進 B 軌 Q1-Q2 → 收集資料 → Q3-Q4 給個人化回饋

### 銜接點的 code 流程

1. 用戶傳「菜單」→ route.js:141 `matchKeyword(text)` 命中
2. route.js:154 呼叫 `rule.handler(userId)` → 回覆 template（末尾「問幾公斤」）
3. 用戶看到後回「50.9 47」→ 下一則訊息進 webhook
4. 不是 4 碼代碼、不是關鍵字 → 進 TEST_MODE gate（route.js:162）
5. **TEST_MODE=true**：非白名單被擋 → 靜默 → 漏斗斷
6. **TEST_MODE=false**：進 handleTextMessage → handleConversationPath → extractWeights 抓數字 → 進 Q2

### 銜接鉤子的 template 位置

- 關鍵字規則：`lib/keywords.js`（先讀 DB `official_settings`，fallback 到 code）
- 回覆文案：`official_settings` 表（後台可編輯）
- A 軌 handler：`app/api/webhook/route.js` handleMenu / handleTrap 等

### 斷點 2：B→Q5 未 wire ⏳ Phase 4.2

- **位置**：Phase 4.2 未實作
- **現象**：用戶走完 Q4 後再傳文字 → stage>=4 fall through → **Bot 靜默**（訊息有進 DB）
- **規模**：會隨 Q4 完成人數增加
- **一休 2026-04-23 判斷**：先接受「Q4 後訊息靜默」，訊息有進 DB 他可人工看。Phase 4.2 上線前不加暫時應答
- **解法**：Phase 4.2 接 Q5 classifier + performQ5Transition（PR #25 helper 已備）

### 斷點 3：handoff notify 目前 push 給一休 + 婉馨 ⚠️ 預期行為

- **位置**：lib/handoff.js:16-19 寫死 NOTIFY_USER_IDS
- **現象**：用戶觸發 handoff（想報名 / 多少錢 / 我老公反對）→ 立即 push 通知
- **2026-04-23 後**：TEST_MODE=false，handoff notify 會開始進一休+婉馨的 LINE。一休已同意
- **解法**：若要調整收件人，改 official_settings 的 `handoff_notify_to`（["yixiu"] / ["wanxin"] / ["yixiu","wanxin"]）

### 斷點 4：/apply LIFF userId 防分享做不到 ⏳ Phase 4.1 / 4.5

- **位置**：LINE 2024 policy 把 LIFF 強制建在 Login channel 下，userId namespace 跟 Messaging API 不同 hash
- **現象**：LIFF `getContext().userId` 在桌面 LINE 是 (none)，手機 LINE 是 Login channel 的 ID（不是 URL userid 那個 Messaging API ID），比對永遠失敗
- **PR #30 修法**：移除 userId 比對 + 移除 isInClient redirect（桌面 LINE 會 loop）→ 現在只做 LIFF init
- **真防護在 Phase 4.1**：/api/apply/submit 驗 HMAC signed URL + ts（擋 URL 造假 / 過期）
- **LINE-to-LINE 分享**：Phase 4.5 觀察期若看到重複 phone/email 再加手機驗證碼
