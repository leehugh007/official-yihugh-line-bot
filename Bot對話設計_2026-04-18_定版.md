## 1. 背景與目標

### 問題
目前官方 Bot 只做「關鍵字比對 → 固定回覆」，不命中就不回覆。結果：
- 用戶沉默後只能靠排程文章，無法根據個人狀況跟進
- 系統不記得用戶聊過什麼、關心什麼、猶豫什麼
- 3-4 小時後用戶回覆的對話接不上

### ROI 驗證
從 2025/8 到 2026/4 的真實數據：
- **官方 LINE 總加入人數 → 約 10% 報名 12 週課程（$11,400）**
- 對比廣告名單（報名率 < 3%），這個漏斗名單品質高很多（做完測驗、領代碼、加 LINE 的高意圖族群）
- **提升 5% 成交 = 每 100 人多 5 萬營收**（單筆課程 1 萬，利潤率高）
- 值得投入 AI + 個人化設計

### 目標
把 10% 報名率提升到 **15%**，途徑是「精準接住高意願族群」。

### 漏斗定位
```
代謝測驗 → 加 LINE → (本 Bot 對話) → 報名 12 週課程 $11,400
                           ↓
                   高意願者 → Handoff → 一休/婉馨手動接
```

**終點是 12 週課程（$11,400），不是阿算**。阿算在本 Bot 的對話中完全不出現。

## 2. 核心設計思路

### Code Gate + AI 判定 + Code 執行（一休熟悉的 pattern）

```
用戶訊息 → Code Gate 過關鍵詞（擋範圍）
         → AI 判定（語意、情緒、路徑分類）
         → Code 執行（套模板、更新 stage、觸發 Handoff）
```

不是讓 AI 瞎聊，是用結構化路徑引導用戶到成交訊號。

### 核心判斷
| 原則 | 做法 |
|---|---|
| 選項式優於開放式 | 用戶跟你還沒深關係，給選項 5 秒能回，開放式 60% 的人不知道怎麼下筆 |
| 代謝重建語言必須埋入 | 一般 AI 寫得出的對話 = 跟直銷/營養單位一樣，沒差異化 |
| 不是所有路徑都要推銷 | 產後路徑 0% 成交，硬推無效 → 改推文章建立信任 |
| 高意願自動 Handoff | 問價格 59% 成交率 — 這種訊號不能讓 AI 繼續聊，直接轉人工 |
| 簡潔優先 | 對話簡潔是核心，不堆砌內容 |

### 婉馨資料給了什麼方向（她的貢獻）

婉馨 2026-04-17 提供的 5,976 筆對話 + 288 位成交反推，產出：

**痛點成交率排名**：
| 痛點 | 提及 | 成交率 | 洞察 |
|---|---|---|---|
| 外食族 | 11 人 | **45%** | 最高，需要「簡單能執行」 |
| 吃過減肥藥 | 17 人 | **41%** | 試過藥但想找健康方法 |
| 健檢紅字 | 44 人 | **36%** | 量大、有健康急迫感 |
| 試過 168 | 16 人 | 25% | 方法失敗 |
| 試過代餐 | 12 人 | 25% | 花過錢但沒效 |
| 瘦瘦針 | 11 人 | 9% | 偏觀望 |
| 復胖 | 238 人 | 4% | 量最大但多數猶豫 |
| 體脂焦慮 | 258 人 | 2% | 量最大但多半只是焦慮 |
| 產後 | 8 人 | **0%** | 帶小孩時間有限，不急著賣 |

**猶豫原因成交率**：
| 訊號 | 成交率 | 洞察 |
|---|---|---|
| 問「價格」 | **59%** | 最強成交訊號 |
| 提到「家人/老公」 | **50%** | 需要支持但有行動意願 |
| 說「怕」 | 44% | 怕失敗但仍在考慮 |
| 問「費用」 | 34% | 人數最多的高意願族群 |
| 「考慮/先了解」 | 21% | 還在觀望 |
| 說「太貴/預算」 | **0%** | 明確拒絕 → AI 應禮貌結束 |

這些數字決定了本設計的路徑優先級和 Handoff 觸發邏輯。

## 3. 對話地圖（Q0 → Q4 完整文案）

### 🔵 Q1 — 破冰（拿完代謝報告後自動推送）

```
報告看完了嗎？
先跟我說一下 — 你現在幾公斤，想瘦到幾公斤？
我看看你狀況，幫你抓個方向。
```

**AI 任務**：抽取 `current_weight` / `target_weight` → 算差距 → 選 Q2 開場模板。

---

### 🔵 Q2 — 接住數字 + 分路徑

**依差距擇一（埋「最後一次 / 瘦一輩子」訊息）**：

**差距 ≤5 公斤**
```
{diff} 公斤不算多。
但關鍵不在這次瘦下來，在瘦完不再回來。
很多人減過很多次，問題是沒有一次是最後一次。
```

**差距 5-15 公斤**
```
想瘦 {diff} 公斤，幅度不小，但做得到。
重點不是這次能瘦幾公斤 —
是瘦完之後，能不能穩穩維持，這次就是最後一次。
```

**差距 >15 公斤**
```
想瘦 {diff} 公斤，幅度很大。
我帶過比你狀況更糟的學員瘦回來過。
關鍵不是一下瘦多少，是代謝底盤要先打好，
這次才是最後一次。
```

**接著問主因（選項本身帶觀念）**：
```
你想開始瘦，主要是為了哪個？

A 健檢紅字、想把數字壓回來
B 以前瘦過又復胖、想結束這個循環
C 產後一直瘦不回來
D 外食族、怎麼吃都搞不定

或有其他狀況直接講，我看看。
```

**選項設計邏輯**：
- A「壓回來」→ 引出治標 vs 治本
- B「結束這個循環」→ 引出瘦一輩子
- D「怎麼吃都搞不定」→ 不是你的問題是方法

**AI 任務**：讀回覆 → 分類到 A/B/C/D/E → 寫入 `path` 欄位。

### 🔴 路徑 A：健檢紅字（36% 成交率）

#### Q3 — 分紅字種類
```
紅字哪幾項？多選 OK，也可以直接回數字。

A 血糖 / 糖化血色素
B 膽固醇 / 三酸甘油脂
C 血壓
D 肝指數

順便講一下 — 醫生有沒有叫你開始吃藥？
```

#### Q4 — 按紅字種類 + 是否吃藥分支

| 子情境 | 回覆 |
|---|---|
| **血糖紅** | 「血糖紅不只是數字，是胰島素已經開始失衡 — 這叫胰島素阻抗（白話講就是身體處理糖分的效率變差了）。繼續下去會越來越難瘦、越吃越餓，身體在發警報。好消息是這可以重建，不用一輩子吃藥。」→ Flex 學員案例（糖化從 7.2 → 5.8） |
| **膽固醇紅** | 「**八成膽固醇是身體自己做的**，不是油吃太多 — 真正的原因是『身體在發炎』。ABC 的 C 就在處理這個。少吃油沒用，要先把發炎降下來。」 |
| **血壓紅** | 「血壓跟代謝綁在一起。代謝重建起來，血壓通常會跟著回正常。吃藥是壓症狀不是解因 — 我很多學員 3 個月停藥的。」 |
| **已吃藥** | 「吃藥壓的是數字，原因還在。有學員 3 個月把紅字拉回正常就停藥 — 他做的不是吃更少，是吃對。蛋白質拉足、發炎降下來、胰島素敏感度回來（白話講就是身體處理食物的效率）。」→ Flex 學員卡片 |
| **未吃藥（初期）** | 「現在是黃金期。早 3 個月處理，跟拖到要吃藥 5 年後再處理，差別非常大。」 |

**結尾統一**：
> 想看怎麼做的嗎？直接回我「了解」就好。

→ 回「了解/怎麼做/費用/報名」→ **Handoff**

### 🟠 路徑 B：復胖（168 試過 25% / 復胖 4%）

#### Q3
```
先問你兩個 — 之前最瘦到幾公斤？維持了多久？
然後是怎麼回來的？

A 停掉那個方法，慢慢就回來了
B 壓力大吃回去
C 就...不知道怎麼回事回來了

直接說也 OK。
```

#### Q4

| 子情境 | 回覆 |
|---|---|
| **A 停掉就回** | 「這不是你沒意志力 — 是節食把你的代謝壓低了。你一回到正常吃，體重自然就回來，因為代謝還沒修好。很多人減肥成功過，但只成功過一次，就是這個原因。ABC 的做法是先把代謝重建起來再瘦，瘦完正常吃也不會回。這才是最後一次。」 |
| **B 壓力型** | 「壓力一大就想吃，這不是你沒意志力 — 是荷爾蒙在影響你。壓力大皮質醇（壓力荷爾蒙）會升高，身體就開始想要高糖高油的東西，這是生存機制。我們要解決的不是叫你忍 — 是讓壓力大的時候，荷爾蒙也能正常運作，不用一直靠意志力撐到最後爆發。」 |
| **C 不知道怎麼回** | 「很多人最後都會問一個問題 —『我也沒吃特別多，怎麼就默默胖回去了？』不是你吃太多，是代謝已經開始失衡了。胖回來，其實是身體在發訊號。要學的不是少吃，是把代謝重建起來。」 |

**結尾統一**：
> 想聽這套方法嗎？

→ Flex：[想了解 / 先看學員故事]

### 🟣 路徑 C：產後（34 次提及 / 0% 成交 ⚠️）

**本路徑策略**：不急著導價格，**先推文章建立信任，等她主動回來再聊**。AI 任務是把她留在生態系，不是成交。

#### Q3
```
生完多久？小孩現在多大？
你現在最影響你的是哪一點？

A 時間真的不夠
B 知道該吃但不知道怎麼開始
C 還在哺乳不敢亂動

直接講也可以。
```

#### Q4 — 推文章，不推方案

| 子情境 | 回覆 |
|---|---|
| **A 時間** | 「媽媽帶小孩卡時間，我懂。產後瘦不回來不是你懶 — 是生完荷爾蒙還沒調回來，加上長期睡不飽。身體在這個狀態下，硬用以前的方法根本行不通。先看一下溫溫的故事，跟你狀況很像。看完有感覺我們再聊。」→ Flex：溫溫故事卡片 |
| **B 方法** | 「產後的身體跟產前完全不一樣 — 荷爾蒙還沒完全調回來，身體對食物的反應也變了，白話講就是吃一樣的份量更容易胖。硬用以前的方法一定撞牆。先看這篇，產後代謝要注意的三件事。」→ Flex 文章 |
| **C 哺乳** | 「哺乳期不能亂節食 — 奶量跟營養都要顧好。有專門給哺乳媽媽的方法，不是少吃，是吃對。先看這篇。」→ Flex 文章 |

**結尾統一**：
> 看完有感覺再回來跟我說，不急。

**產後路徑 Handoff 觸發放寬**：等她看完文章主動回來（不用命中關鍵字）就轉人工。

### 🟡 路徑 D：外食族（45% 成交最高）

**本路徑策略**：**先聽她怎麼吃，我幫你看**。不是「我告訴你你有問題」。

#### Q3
```
你平常大致怎麼吃？先簡單說一下是哪種型態：

A 三餐都外面（超商 / 便當 / 小吃 / 早餐店）
B 早午外食、晚上自己煮
C 外食為主但有時自煮
D 其他

選好後跟我具體講一下你常吃什麼 —
早餐、午餐、晚餐舉幾個例子，
我幫你看看有什麼可以調整的。
```

#### Q4 — AI 讀用戶實際餐點，找血糖問題，帶出 SOP

**AI prompt 骨架**：
```
讀用戶分享的餐點 → 找一個最明顯的血糖穩定相關問題
→ 白話指出（避免術語或術語配白話翻譯）
→ 帶到「我們教的不是菜單，是怎麼在各種場景選對組合」
→ 延伸到聚餐 / 旅遊 / 應酬都能用
```

**範例（早餐三明治配奶茶 + 午餐便當）**：
```
你早餐三明治配奶茶，
剛好是血糖起伏最大的組合 —
吃完血糖飆高，兩小時後掉下來，
就又餓又累、想吃甜的。

這不是你嘴饞，是血糖在玩你。

我們會教的不是一套菜單 —
是不管你在小 7、早餐店、還是自助餐，
都能挑對組合，讓血糖穩住。
吃飽、不會一下又餓、不會暴飲暴食。

學會這套，聚餐、旅遊、應酬都用得上 —
這輩子都外食也能瘦。
```

→ Flex：[想學這套 / 先看課程細節]

#### AI 引導策略（關鍵：用戶一直不命中時怎麼辦）

**原則**：AI 把用戶**主動引導到命中關鍵字的方向**，不放任開放對話。

**引導句庫**：

| 用戶亂回的內容 | AI 判斷 | 引導回覆 |
|---|---|---|
| 「我就是懶」 | 接近 B 或動機 | 「懶通常不是原因 — 很多時候是身體已經累了。你之前有試過什麼方法嗎？還是最近健康有什麼狀況？」 |
| 「就是想變好看」 | 接近動機 | 「想變好看沒問題。但我問你 — 你是想瘦一陣子穿漂亮衣服，還是想瘦一輩子不用每年再來一次？你平常常外食嗎？有健康警訊嗎？」 |
| 「我沒什麼特別的」 | 完全發散 | 「那我換個問法 — 你最近一次量體重，比去年同期是胖還是瘦？為什麼？」 |
| 「我只是想了解」 | 觀望 | 「了解可以，我問你兩個問題幫你釐清 — 你現在幾公斤，想瘦到幾公斤？瘦了之後最想改變生活裡哪一塊？」 |

**連續兩次還不命中 → Fallback**：
```
不急著回答，先看一下這兩個故事，你看完再跟我說。
[Flex：溫溫故事 + 外食族典型案例]
```

## 4. Schema 設計

> **2026-04-18 yi-challenge + yi-safe-change 補丁**：在定版 8 欄位基礎上補 4 個「避坑 + 量測」欄位。不改對話流程、不改路徑，只補防禦補丁。
> 補丁來源：雙 agent（Opus 4.7）審查指出 `ai_tags` JSONB 是阿算 journey 翻版（2026-04-02 停用，缺失效機制會累積過時資訊）；`path_stage` 無 timeout 會永久冷凍用戶；10%→15% 北極星需要成交歸因欄位才量得出來。
> 補丁邏輯類似 `is_active DEFAULT false`（當初從 Drip 04-04 事故補的防禦），屬避坑補強不是重新設計。

### 擴充 `official_line_users`（新增 12 個欄位 = 8 核心 + 4 補丁）

```sql
ALTER TABLE official_line_users
  -- 核心欄位（原定版 8 個）
  ADD COLUMN current_weight NUMERIC,             -- 目前體重
  ADD COLUMN target_weight NUMERIC,              -- 目標體重
  ADD COLUMN path TEXT,                          -- 'healthCheck' | 'rebound' | 'postpartum' | 'eatOut' | 'other' | NULL
  ADD COLUMN path_stage INTEGER DEFAULT 0,       -- 0=未進 / 1=Q1 / 2=Q2 / 3=Q3 / 4=Q4 已導向
  ADD COLUMN last_user_reply_at TIMESTAMPTZ,     -- 用戶最後主動回覆時間
  ADD COLUMN ai_tags JSONB DEFAULT '{}',         -- {痛點:[], 猶豫:[], 意願:'high|medium|low', 關注:[]}
  ADD COLUMN handoff_triggered_at TIMESTAMPTZ,   -- 專人介入觸發時間
  ADD COLUMN handoff_reason TEXT,                -- 'asked_price' | 'asked_family' | 'high_intent' | 'postpartum_returned' | 'manual'
  -- 避坑補丁（2026-04-18 yi-challenge）
  ADD COLUMN ai_tags_updated_at TIMESTAMPTZ,     -- ai_tags 更新時間。抄阿算 insights 14 天過期機制，Phase 4 AI 重跑時超過 14 天重估，避免 journey 重生
  ADD COLUMN path_stage_updated_at TIMESTAMPTZ,  -- path_stage 更新時間。Phase 3 cron 24h 檢查 stage 2/3 停 > 7 天無互動 reset 0，避免永久冷凍
  -- 北極星量測（10%→15% 成交歸因）
  ADD COLUMN enrolled_from_path TEXT,            -- 用戶升級 tags='已報名減重班' 時 snapshot 當時 path
  ADD COLUMN enrolled_at TIMESTAMPTZ;            -- 成交時間
```

**Phase 3 實作必包含的避坑機制**（寫進指揮中心待辦，不是選配）：
1. `ai_tags` 寫入一律走 `lib/users.js` 的 `updateAiTags()` helper，**禁用 SQL jsonb_set**（Awear 2026-04-14 全滅教訓）
2. `ai_tags` 清理：handoff 成功後清 `意願:'high'` / stage 變更重估痛點 / `_removes` 連動清
3. `path_stage` timeout cron（24h）：stage 2/3 + `path_stage_updated_at` 超過 7 天 + 無互動 → reset 0
4. Webhook 查 reply_templates 加 in-memory cache 60s TTL（避免打阿算共用 Supabase 熱路徑）
5. reply_templates 啟用前 FlexPreview 預覽 + cron 二層 placeholder 防呆（Drip 設計決策 #15）
6. tags 升級「已報名減重班」時同步寫 `enrolled_from_path` + `enrolled_at`（成交歸因）

### 新表 `official_chat_history`（對話記錄）

> **2026-04-18 執行順序調整**：chat_history 表從 Phase 1 移到 **Phase 3** 跟 webhook 邏輯一起上。
> 原因：`path_at_time` / `stage_at_time` 是快照欄位，在 Phase 4 AI 上線前都會是 NULL（沒 code 寫進去），中間 2-3 週空表期白寫 I/O、歷史也補不回。綁 webhook 一起建，上線即有資料。

```sql
CREATE TABLE official_chat_history (
  id BIGSERIAL PRIMARY KEY,
  line_user_id TEXT NOT NULL REFERENCES official_line_users(line_user_id),
  role TEXT NOT NULL,                           -- 'user' | 'assistant'
  message TEXT NOT NULL,
  path_at_time TEXT,                            -- 當時走哪條路徑（快照）
  stage_at_time INTEGER,                        -- 當時第幾步（快照）
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_official_chat_history_user_time
  ON official_chat_history(line_user_id, created_at DESC);
```

### 新表 `official_reply_templates`（模板，後台編輯）

```sql
CREATE TABLE official_reply_templates (
  id TEXT PRIMARY KEY,                          -- 'q1_init' / 'q2_weight_small' / 'path_a_blood_sugar' ...
  path TEXT,                                    -- 'healthCheck' | 'rebound' | ... | NULL（通用）
  stage INTEGER,                                 -- 0 | 1 | 2 | 3 | 4
  condition TEXT,                               -- 'weight_diff_small' | 'blood_sugar' | NULL
  message_template TEXT NOT NULL,               -- 含 {current}{target}{diff} 等變數
  buttons JSONB DEFAULT '[]',                   -- Flex 按鈕 [{label, url_or_postback, linkId}]
  image_url TEXT,                                -- Flex hero 圖
  is_active BOOLEAN DEFAULT false,               -- 預設 false（04-04 placeholder 事故教訓）
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

**為什麼這樣設計**：
- 欄位 `path` + `stage` + `condition` 三層索引，一個 query 找到對應模板
- `is_active` 預設 false — 承襲 04-04 Drip 事故教訓
- 模板骨架寫死在 code，但文字/圖/按鈕在後台可改
- AI 生成用 `message_template` 的變數（{current}{target}{diff}{user_meal}）

## 5. Handoff 觸發條件

### 立即觸發（LINE notify 推一休 + 婉馨）

```
path_stage >= 2
AND 命中任一：
  - 「價格 / 費用 / 多少錢 / 怎麼報名 / 開課 / 下一期」（高意願關鍵字）
  - 「老公 / 先生 / 家人 / 老婆 / 一起」（家人路徑 50% 成交）
```

### 後台紅點不即時推（低頻）
```
AI 判定 ai_tags.意願 = 'high' 但沒命中關鍵字
```

### AI 禮貌結束（不再追）
```
命中「太貴 / 沒預算 / 沒錢 / 先不用」 → intent_level = 'low'
AI 回：「了解，那就先不推銷。如果以後想重新了解可以再找我。」
```

### 產後路徑特殊處理
```
Q4 推完文章後，用戶主動回覆（不用命中關鍵字）→ Handoff
```

### 通知格式（LINE notify）
```
🔴 高意願用戶

第 3 步：問了「學費多少」
代謝類型：rollerCoaster

→ 開對話：[後台連結]
```

## 10. 重要決策紀錄（為什麼這樣設計）

> 本節記錄 2026-04-18 討論中敲定的關鍵決策。未來有疑問「為什麼這樣做」先看這裡。

### 決策 1：問公斤取代選項題
**背景**：一休以前試過純分類選項題，回覆率低。後來改成「想瘦幾公斤？我幫你看」→ 回覆率高。
**理由**：每個想瘦的人心裡都有個數字，零摩擦 + 有 open loop。
**影響**：Q1 破冰用問公斤，Q2 才分類。

### 決策 2：代謝重建語言必須埋入
**背景**：第一版 AI 回覆被一休批評「跟一般營養師/直銷/AI 大同小異」。
**理由**：沒差異化 = 沒說服力。
**影響**：關鍵詞「瘦一輩子 / 最後一次瘦身 / 吃飽不用餓肚子」、「代謝失調 / 荷爾蒙 / 身體發炎 / 胰島素」要分散埋進對話，不堆砌。

### 決策 3：導向 12 週課程，不是阿算
**背景**：原本設計混入阿算，但官方 Bot 目標是課程招生。
**理由**：官方 LINE 從去年 8 月到今年 4 月的 ROI 驗證 = 10% 報名課程。阿算是另一條漏斗。
**影響**：本 Bot 對話中阿算完全不出現。終點是 12 週課程 $11,400。

### 決策 4：產後路徑特殊處理（不急著導價格）
**背景**：數據顯示產後 8 人 / 0 人成交（0%）。
**理由**：媽媽帶小孩時間有限 + 產後身體特殊，硬推無效。需要先建立信任資產。
**影響**：產後路徑 Q4 只推文章，不導課程。Handoff 觸發放寬（她回來主動回就算）。

### 決策 5：外食族改「先聽你說」
**背景**：原設計「我告訴你你有問題」，一休指出應該是「你說，我幫你看」。
**理由**：用戶不一定知道自己有問題，甚至可能以為吃法是對的。「先聽再給」是教練口吻，不是推銷口吻。
**影響**：Q3 改成「你跟我說平常怎麼吃」，Q4 AI 讀實際餐點給回饋。

### 決策 6：不是意志力是荷爾蒙（主體修正）
**背景**：第一版寫「這不是意志力」，一休糾正應該是「這不是**你沒意志力**」。
**理由**：前者是在談方法，後者是在替用戶脫罪。用戶要的是後者。
**影響**：路徑 B 所有回覆主體改成「你」，不談「意志力」當籠統概念。

### 決策 7：八成膽固醇是身體自製
**背景**：第一版寫「九成」，一休糾正是「八成」。
**理由**：美國心臟協會、哈佛健康學會、台灣衛教主流說法 = 80%。
**影響**：路徑 A 膽固醇子情境文案數字更正。

### 決策 8：Code Gate + AI 判定 + Code 執行
**背景**：純 AI 生成失控風險高，純 Code 關鍵詞又不夠彈性。
**理由**：一休在婉鈞膝蓋事件驗證過這個 pattern — 關鍵詞寧多勿少、Code Gate 擋範圍、AI 做語意、Code 執行動作。
**影響**：整個架構的技術底層。

### 決策 9：後台分層三個 Tab
**背景**：婉馨沒 Supabase 權限、對後台資料庫操作不熟、AI 協作也比較弱。
**理由**：成交關鍵流程不能讓不熟悉的人動後端。她的價值在實際使用回報。
**影響**：骨架 code 寫死、內容層全部在後台。婉馨不碰 Supabase 和程式碼。

### 決策 10：選項式優於開放式
**背景**：用戶跟 Bot 還沒深關係，一次講太多不切實際。
**理由**：給選項 5 秒能回、完全開放 60% 不知道怎麼下筆。
**影響**：Q2/Q3 都提供 A/B/C/D 選項 + 保留「其他狀況直接講」。
