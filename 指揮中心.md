## 踩坑紀錄

### 2026-04-04 — Drip placeholder 發給 122 人

- **影響**：6 篇排程文章內容是「待填入訊息內容」+ example.com 連結，cron 從 4/1 到 4/4 自動發給 122 人（全部用戶的 93%）
- **根因**：`is_active` 預設 `true`，排程文章建好就自動啟用。沒有「填內容→確認→啟用」的流程，cron 不分內容直接發
- **修復**：(1) 全部停用 (2) is_active default 改 false (3) cron 加 placeholder 防呆 (4) 新增啟用驗證 API（內容非空/非 placeholder/連結非 example.com）(5) 後台加啟用前預覽確認
- **教訓**：自動化系統的預設值要保守（default off），內容型功能必須有「草稿→確認→上線」流程。建好 ≠ 可用

### 2026-04-01 — CRON_SECRET 未設定，排程推播全部 401

- **影響**：Vercel Cron 每 10 分鐘觸發全部 401，排程推播無法自動發送。手動觸發（帶 ?secret=）可以，造成「手動能跑、自動不行」的假象
- **根因**：vercel.json 有 cron 設定但 Vercel 環境變數沒有 CRON_SECRET。Cron 帶 `Authorization: Bearer ${CRON_SECRET}`，server 端比對 `undefined` 永遠失敗
- **修復**：Vercel 環境變數加 CRON_SECRET + 重新部署
- **教訓**：設 cron 時環境變數要同步設好，不要假設「之後再設」

### 2026-04-01 — VERCEL_URL 回傳 preview URL

- **影響**：cron 裡 fetch 自己的 admin API 失敗，排程推播不會被自動發送
- **根因**：`process.env.VERCEL_URL` 回傳 deployment-specific preview URL，不是 production URL
- **修復**：提取 `lib/push.js` 共用邏輯，cron 直接呼叫函數不繞 HTTP
- **教訓**：Vercel serverless 裡不要 fetch 自己的 API，直接 import 函數呼叫

### 2026-04-01 — datetime-local 不帶時區，排程差 8 小時

- **影響**：排 19:15 的推播實際 03:15（隔天）才發
- **根因**：`<input type="datetime-local">` 回傳不帶時區字串，存 Supabase TIMESTAMPTZ 被當 UTC
- **修復**：DateTimePicker24 元件統一帶 +08:00
- **教訓**：Vercel 跑 UTC，所有時間必須明確帶時區
