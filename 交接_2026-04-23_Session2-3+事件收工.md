## 關鍵決策 & 教訓

### 1. Landing copy：raw 原話 > polish narrative

v1 → v4 四輪 iteration 一休都說「還是很爛」，直到 v4 **用一休原話當骨架**（「喝水都會胖 / 討厭胖的自己 / 真正最好的瘦身是你不覺得在減肥 / 綻放的光芒」）一休才說「可以好多了」。

**學到**：一休回「還是很爛」時不要再 polish，放下自己的手、把他的原話當骨架做最小編輯銜接。

### 2. stage=0 保持靜默保護新用戶

A 軌 matchKeyword 命中時才 upgrade stage=0→1（精準鎖定「經過 A 軌」的用戶）。純加好友打招呼的用戶 stage 仍 0 → 繼續靜默，不被硬塞「要兩個數字」。

### 3. 雙人早鳥走人工不進 form enum

Ch.5 顯示 anchor card（$3,333/人、橘色虛線）+ CTA「回 LINE 找 fifi 團報」。避免改 schema + RPC + /api/apply/submit 白名單。

### 4. stage=4 bridging 臨時，Phase 4.2 拿掉

PR #40 / #41 的 stage=4 bridging 有 `TODO Phase 4.2` 註解。Q5 classifier wire 上線後，stage=4 改走 Q5 classifier 分流（參考契約 v2.4 Ch.5.1a 被動軌規格）。

### 5. Vercel 部署不是即時

PR merge 不等於 prod 部署。Vercel build queue 可能拖 30+ 分鐘。改動部署前觀察 `mcp__claude_ai_Vercel__list_deployments` 或 `vercel ls` 看實際 READY 時間。**部署期空窗打到舊版 code** 是常見的 race。

### 6. Write tool 回 success 不等於檔案變更

2026-04-23 有 2 次 Write tool 回 "File updated successfully" 但 git diff 實際無變動（page.js）。**Write 之後一律用 grep 關鍵字 + `wc -l` 驗證實質內容**。build pass ≠ 內容對。

### 7. PR 多時需要編號辨識

一休 review PR #36/#37 時把「close」跟「merge」弄反，差點讓 v4.2 沒上 prod。PR title 要更具區分度，或用 squash merge description 帶動作暗示。
